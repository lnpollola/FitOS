## Why

FitOS currently splits its analytical surface between a **data-dense dashboard** (recently enhanced with Strava-style summary panels) and a **power-user analytics view** with raw trend charts. There is no middle tier that converts weeks of accumulated data into a **narrative of patterns** ("tu mejor semana en 3 meses", "los lunes corres 28% más", "HRV 12% sobre tu promedio"). The data is already there — `sport_activities` (with `distance_km` and `duration_minutes`), `activity_days` (with `sleep_hours` and `sleep_deep/rem/light`), `weight_entries`, `measurement_sets`, and HealthSync tables (`hrv`, `resting_heart_rate`, `vo2_max`) — but it lives in 4 different views and 2 different databases. A user asking "what's actually happening with my training and recovery over time?" has to assemble the answer manually. This change adds a dedicated **`insights` view** between Panel and Tendencias, surfacing the patterns the data already encodes: a year-in-motion heatmap, day-of-week rhythm, sport distribution, recovery composite, weight velocity, waist-to-hip ratio, and auto-generated insight cards. Code name in English (`insights`); UI strings remain in Spanish.

## What Changes

- **New `insights` view**: dedicated view positioned between `dashboard` (Panel) and `analytics` (Tendencias) in the sidebar's INICIO section. Single-page scroll layout with 7 distinct sections: (1) year-in-motion heatmap, (2) day-of-week histogram, (3) sport distribution donut, (4) recovery score composite, (5) weight velocity line, (6) waist-to-hip ratio card, (7) auto-generated insight cards.
- **Year-in-motion heatmap**: 53 weeks × 7 days SVG grid showing total daily training minutes. Each cell shaded on a 5-step scale (`var(--moss-mist)` → `var(--moss-ink)`) by minutes bucket. 30-second render budget; pure SVG with 371 cells max. Hover/tap reveals tooltip with date and minutes.
- **Day-of-week histogram**: 7-bar chart (L-D) showing total minutes and session count per weekday. Highlights the user's "best day" (most minutes) with a small badge.
- **Sport distribution donut**: Chart.js doughnut chart showing share of total minutes per `sport_type` over the last 90 days. Center label shows total hours. Legend lists each sport with minutes, sessions, and percentage.
- **Recovery score composite**: 0–100 score combining 3 personal-baseline-normalized signals: HRV (weight 0.4), RHR (weight 0.3, inverted), sleep (weight 0.3). Each sub-signal shows its own mini-meter with baseline annotation. Composite uses a 7-day rolling window and is color-coded (`low` < 40 / `moderate` 40–70 / `high` > 70).
- **Weight velocity line chart**: 28-day rolling weight-delta rate (kg/week) over the last 6 months. Reference band at the user's `target_pace` from settings. Annotated with PR-weight marker (lowest ever).
- **Waist-to-hip ratio card**: latest WHR from `measurement_sets` with the WHO/OMS zone classification (Hombres: <0.90 bajo, 0.90–0.99 moderado, ≥1.00 alto; Mujeres: <0.80 bajo, 0.80–0.84 moderado, ≥0.85 alto). Color-coded by zone. Sparkline of last 90 days of WHR.
- **Auto-insight cards**: a feed of 5–8 short text cards generated by deterministic heuristics from the same data (e.g. "3 semanas consecutivas con más de 5h de entrenamiento — tu mejor racha desde marzo", "HRV 12% sobre tu promedio de 30 días", "Llevas 6 días sin actividad — ¿descanso o rutina perdida?"). Each card has an icon, a 1-line text, and a "ver detalle" link that navigates to the relevant view.
- **IPC additions**: 7 new handlers (`db:getYearInMotion`, `db:getDayOfWeekStats`, `db:getSportDistribution`, `db:getRecoveryScore`, `db:getWeightVelocity`, `db:getWHR`, `db:getAutoInsights`) that compute derived KPIs from existing tables. No new tables.
- **Renderer additions**: new `src/renderer/views/insights.js` (~600 lines) with `init()` function following the existing per-view pattern. New derived-KPI helpers in `src/renderer/utils/kpi-derivation.js` (~120 new lines: `recoveryScore()`, `weightVelocity()`, `whrZone()`, `dowPattern()`, `sportDistribution()`) — all pure functions, no SQL.
- **IPC handler additions**: new `src/main/handlers/insights-handlers.js` (~300 lines) consolidating the 7 derived-KPI queries. Mirrors the modular-handlers refactor pattern (separate file per domain).
- **Sidebar update**: add `<li data-section="inicio"><button class="nav-item" data-view="insights">…</button></li>` between `dashboard` and `analytics`. Icon: `sparkles` (Lucide). Spanish label: "Patrones".
- **App.js routing**: register `insights: initInsights` in the `views` map. Add `'insights': 'sparkles'` to the `iconMap` in `renderNavIcons()`.
- **Locales**: new keys under `strings.insights.*` (~80 keys: section titles, empty states, recovery zone labels, WHR zone labels, auto-insight text templates with interpolation, "actualizar" / "compartir" labels). All in Spanish.
- **CSS**: extend `src/renderer/styles/main.css` with `.insights-*` classes (~200 lines) using the existing organic-aesthetic tokens. New heatmap cells, donut center label, recovery meter, auto-insight card.
- **Tests**: new unit tests in `tests/unit/kpi-derivation.test.js` (~15 new tests for `recoveryScore`, `weightVelocity`, `whrZone`, `dowPattern`, `sportDistribution`, `generateAutoInsights`). New smoke test in `tests/smoke/insights.test.js` (~8 tests: view renders, all 7 sections mount, skeleton → data flow, error state on IPC failure, empty state for fresh DB). Target: existing 180 tests + 23 new tests = 203/203 passing.
- **No breaking changes**: new IPC handlers are additive; new view is additive; existing sidebar order, dashboard layout, and all other views continue to function identically. The `dashboard-health-metrics` spec gets a single "ADDED Requirement" noting that the `insights` view now exists as a navigational companion (no behavior change to dashboard itself).
- **No schema changes**: zero new tables, zero new columns. All KPIs derive from existing data.
- **No new npm dependencies**: heatmap is hand-rolled SVG, donut reuses Chart.js, line chart reuses Chart.js.

## Capabilities

### New Capabilities

- `year-in-motion-heatmap`: Render a 53-week × 7-day SVG heatmap of total daily training minutes from `sport_activities`, with a 5-step color scale, hover tooltips, and ISO week + day-of-week labels.
- `day-of-week-pattern`: Compute and display total minutes and session count per weekday from `sport_activities`, with the user's "best day" highlighted and a 7-bar Chart.js histogram.
- `sport-distribution-donut`: Visualize share of total training minutes per `sport_type` over the last 90 days as a Chart.js doughnut chart with center total-label and per-sport legend (minutes, sessions, %).
- `recovery-score-composite`: Compute a 0–100 recovery score from HRV (40%), resting heart rate (30%, inverted), and sleep (30%) over a 7-day rolling window, normalized against each signal's 30-day personal baseline. Render the composite with color-coded zone and three sub-meters.
- `weight-velocity-chart`: Compute a 28-day rolling weight-delta rate (kg/week) from `weight_entries` over the last 6 months, displayed as a Chart.js line with a reference band at the user's `target_pace` and an annotated "lowest ever" marker.
- `waist-hip-ratio-card`: Compute latest WHR from `measurement_sets` (waist / hips), classify into the WHO/OMS zone based on user sex from `user_profile`, color-code by zone, and show a 90-day sparkline.
- `auto-insight-cards`: Generate 5–8 deterministic insight text cards from the underlying KPIs using template-based heuristics ("best week in N months", "HRV deviation", "rest day streak", "sport variety score", "weight direction match", etc.). Each card has an icon, a 1-line text, and a navigation link.
- `insights-view`: New top-level view positioned between `dashboard` and `analytics` in the sidebar's INICIO section. Renders all 7 sections in a vertical scroll layout with shared skeletons, concurrent IPC loading via `Promise.allSettled`, and a date-range selector (90d / 6m / 1y) that gates the heatmap, histograms, and velocity chart (recovery and WHR are point-in-time and ignore the selector).

### Modified Capabilities

- `dashboard-health-metrics`: Add a single ADDED Requirement noting that the dashboard's navigational context now includes a companion `insights` view (positioned between dashboard and analytics in the sidebar). The dashboard's own layout, panels, and behavior are unchanged. This is a cross-link rather than a behavior modification — required only so the spec accurately reflects the new top-level view in the app.
- `sidebar-sections`: Add the `insights` nav item to the INICIO section, between `dashboard` (Panel) and `analytics` (Tendencias). The section's expand/collapse, active-view pinning, and localStorage persistence behavior is unchanged. The new item uses the existing pattern: `data-section="inicio"`, `data-view="insights"`, Lucide `sparkles` icon, label "Patrones".
- `spanish-ui`: Add new locale namespace `strings.insights.*` (~80 keys). All in Spanish. Includes: section titles, recovery zone labels (Bajo/Moderado/Alto), WHR zone labels (Bajo/Moderado/Alto/Recomendado), sport variety labels, weight velocity direction labels, auto-insight text templates with `{value}` placeholders, empty states per section, date-range selector labels, "Actualizar" button label, "Ver detalle" link label.
- `iconography`: Register the Lucide `sparkles` icon (insights nav item) and `lightbulb` icon (auto-insight card icon) as tree-shaken imports in `src/renderer/utils/icons.js`. No sport-icon changes in this change (Phase 1 already covered them).

## Phased delivery

This change is **Phase 2** of a multi-phase plan to progressively deepen the dashboard's summary layer. Each phase is a self-contained `/opsx-propose` change with its own specs, tasks, and tests, and can be implemented, tested, and deployed independently.

| Phase | Change name | Scope | Status |
|---|---|---|---|
| **1** | `panel-ux-ui-kpis-summarized` | Strava-style summary panels on the dashboard. 126/126 tasks. Archived 27 Jun 2026. | **Done** |
| **2 (this change)** | `summary-insights-view` | A new `insights` view between `dashboard` and `analytics` with year-in-motion heatmap, day-of-week pattern, sport distribution, recovery score, weight velocity, WHR, and auto-insight cards. Code name in English; UI strings in Spanish. | **Proposing** |

| **4 (later)** | `goals-tracker` | User-configurable goals (weight target, monthly distance, exercise PR target), progress rings with countdown, achievement badges. | Planned |

**Phase 2 exists to keep the dashboard from becoming too dense.** Phase 1 already added ~400 lines of view code + 270 lines of CSS to the dashboard. The Phase 2 view provides a curated middle tier: more than the at-a-glance dashboard, less than the power-user `analytics` view. It surfaces **patterns over time** — the data the dashboard compresses away in service of "what's happening now".

**Data sources for all 4 phases already exist** in the database. No new IPC layer (just 7 additive handlers), no new tables, no new dependencies. The phases differ only in the renderer layer, the derived-KPI computations, and the IPC payload shapes.

## Why a separate view (not another panel on the dashboard)?

The Phase 1 panels were deliberately narrow — at-a-glance signals that earn their space above the hero card (PR, weekly goal, effort, training log, streak, calendar). The insights in this change are **fundamentally different** in three ways:

1. **Time horizon.** Panels show the current week / current month. Insights show the trailing 90 days / 6 months / 1 year — patterns that don't fit in a single "this week" frame.
2. **Heuristic density.** Panels are pure SQL aggregates (count, sum, min). Insights require compositing 3+ signals into a derived score (recovery = HRV + RHR + sleep), binning daily minutes into a 53-week grid, and generating natural-language text from heuristics.
3. **User mental model.** Panels are "what just happened". Insights are "what does it mean over time". Mixing both on the same screen would dilute each — and the dashboard would exceed 1500 lines of view code.

A separate view with its own URL, its own date-range selector, and its own loading choreography is the cleanest way to give these patterns the space they need.
